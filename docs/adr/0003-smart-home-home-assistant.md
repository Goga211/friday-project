# ADR 0003 — Умный дом: Home Assistant как хаб, Mock — до появления железа

Статус: принято (Phase 4). Дата: 2026-07-03.

## Контекст
Мастер-план (§2.4) оставлял выбор экосистемы умного дома открытым: устройств и хаба нет,
«вероятное направление — Home Assistant, но не зафиксировано». Phase 4 требует рабочего
пути мозг → устройства дома уже сейчас.

## Решение
1. **Абстракция `DeviceController`** (Protocol: list/find/get_state/set_state/run_scene) —
   код агента не зависит от конкретного хаба; реализация выбирается конфигом
   (`FRIDAY_HOME_CONTROLLER=mock|ha`).
2. **Home Assistant — целевой хаб.** Адаптер `HomeAssistantController` — поверх REST API
   (`/api/states`, `/api/services/<домен>/<сервис>`, long-lived token). Обоснование:
   - адаптер пишется **один раз**; новое устройство добавляется в HA, а не в код —
     per-device кода ≈ ноль;
   - HA поддерживает тысячи интеграций (Tuya/Xiaomi/Zigbee/…) — не привязываемся к
     экосистеме устройств до их покупки;
   - HA сам на Python и ставится на тот же Hub-мини-ПК.
3. **`MockController` — по умолчанию**, пока железа нет: in-memory лампы/розетка/сцена с
   честными переходами состояний. Весь путь (мозг → шина → агент → контроллер) работает и
   тестируется без единого устройства.
4. **Риск действий — safe** (решение пользователя): свет/розетки без подтверждений.
   Если появятся замки/климат — риск ужесточается правкой реестра возможностей агента
   (одно поле), архитектура не меняется.

## Последствия
- Живой прогон адаптера HA отложен до появления железа (юнит-тесты — на httpx.MockTransport).
- WebSocket API HA (события, push-обновления состояний) — не сейчас; REST хватает для
  команд и опросов. Пересмотреть при проактивности (Phase 5).
- Сцены/автоматизации создаются в HA и дёргаются как `home_run_scene`.
